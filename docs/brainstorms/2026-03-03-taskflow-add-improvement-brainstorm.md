# /taskflow add の改良ブレスト

**日付**: 2026-03-03
**ステータス**: ブレスト完了

## 何を作るか

`/taskflow add` コマンドのフォローアップフローを改善し、タスク追加後に以下を必ず確認するようにする：

1. **プロジェクト紐付け** — タイトルから既存プロジェクトを推測し確認。推測不可時は一覧から選択。`project_id` が null のまま通過させない
2. **ワークスペース（cmux）作成確認** — タスクに対してセッション+ワークスペースを作成するか確認

## なぜこのアプローチか

- **ポストプロセス方式**を採用。タスク作成自体は即座に完了させ、後から紐付けとワークスペース確認を行う
- タスク追加が最優先という方針と合致。途中中断してもタスクは残る
- API呼び出しが1回増える（PATCH）が、UXの自然さを優先

## キーデシジョン

### 1. プロジェクト推測ロジック

- **ソース**: タスクタイトルのキーワードと既存プロジェクト名のマッチング
- **理由**: Obsidian起点で動作するため、cwdのディレクトリパスは参考にならない
- **フロー**:
  - GET /api/projects で一覧取得
  - タイトルに含まれるキーワードとプロジェクト名を照合
  - マッチしたら「プロジェクト『X』に紐付けますか？」と確認
  - マッチしなければプロジェクト一覧を提示して選択させる
  - いずれの場合もnullのまま通過させない（明示的に「未分類」を選ぶのはOK）

### 2. ワークスペース作成確認

- **タイミング**: プロジェクト紐付けの後
- **フロー**:
  - 「このタスクでワークスペースを作成しますか？」と確認
  - Yesの場合: セッション作成（POST /api/sessions）→ taskflow-cmux start
  - Noの場合: スキップ

### 3. フロー全体像

```
1. タスク作成（POST /api/todos）← 最優先・即座
2. プロジェクト推測 or 選択 → PATCH /api/todos/:id で紐付け
3. ワークスペース作成確認 → セッション + cmux 起動
```

## スコープ

- **対象**: `/taskflow add` サブコマンドのみ
- **対象外**: `/taskflow context`, `/taskflow session start` 等の他サブコマンド（将来の改善対象）
- **変更箇所**: SKILL.md のスキル定義のみ（API・フロントエンドの変更なし）

## Resolved Questions

1. **「未分類」は不可** — 必ず既存プロジェクトのいずれかに紐付ける。該当なしの場合は新規プロジェクト作成を促す
2. **one-shotモードでも確認する** — `--project X` で指定済みでも「プロジェクト『X』でよいですか？」と確認を挟む
