# Brainstorm: 作業セッション（Work Sessions）

**Date:** 2026-02-27
**Status:** Draft

## What We're Building

複数PC・複数プロジェクトで並行作業するときの「コンテキスト喪失」問題を解決する**作業セッション管理機能**。

### 課題

1. 4つほど並行作業していると、各作業がどこまで進んだか把握しづらい
2. 作業を切り替えたとき、前回の進捗・意思決定の経緯を思い出すのに時間がかかる
3. PC1/PC2/PC3で別プロジェクトを進め、さらにプロジェクト内でも複数タスクが並行

### 解決策

新概念 **`work_session`** を導入。各セッションが1つの並行作業ストリームを表す。

- `work_session`: タイトル、project紐付き（任意）、ステータス（active/paused/done）
- `session_log`: セッションに紐づく自由テキストのログエントリ（進捗、決定、次のステップなど）
- 既存の `todo` をセッションに紐付け可能

## Why This Approach

### 「作業セッション」モデルを選んだ理由

- **B案（タスクメモ拡張）** は既存todoの拡張でシンプルだが、並行作業の「全体像を俯瞰する」体験が弱い
- **C案（プロジェクトノート）** はprojectの強化だが、1プロジェクト内に複数作業ストリームがある問題を解決できない
- **A案（作業セッション）** は新概念だが、並行作業を第一級市民として扱い、俯瞰とコンテキスト復帰の両方に強い

### ユーザー体験

1. **俯瞰**: Taskflow Webを開くと、アクティブなセッション一覧が見える。各セッションの最新ログが表示される
2. **復帰**: Claude Codeで `/taskflow context` を実行すると、そのプロジェクトのセッション情報を取得。「前回どこまでやったか」「次のステップは何か」がすぐわかる
3. **記録**: 作業の切れ目でClaude Codeが `/taskflow context` で自動的にログを送信。手動でもWeb UIからメモを追記可能

## Key Decisions

1. **新テーブル2つ追加**: `work_sessions` + `session_logs`
2. **セッションのステータス**: `active` / `paused` / `done` の3状態
3. **ログは自由テキスト**: マークダウン対応。構造化は後から考える
4. **project紐付きは任意**: セッションはprojectに属してもいいし、独立でもよい
5. **todoとの紐付け**: todoに `session_id` を追加（任意）
6. **CLI連携**: 既存 `/taskflow` skill に `context` サブコマンドを追加
7. **ダッシュボード**: フロントエンドにセッション一覧ビューを追加

## Scope

### In Scope (MVP)

- `work_sessions` テーブル + CRUD API
- `session_logs` テーブル + 追加/一覧 API
- フロントエンド: セッション一覧、セッション詳細（ログ閲覧）、ログ追加
- `/taskflow context save` - 現在のセッションにログを追加
- `/taskflow context load` - セッションの最新コンテキストを取得

### Out of Scope (後回し)

- todoとセッションの紐付け（session_id）
- セッションの自動アーカイブ
- PC名/デバイス情報の記録
- AIによるサマリー自動生成
- 通知/リマインダー

## Resolved Questions

1. **セッション一覧の配置** → 新しいダッシュボードビューを追加。アクティブセッション + 今日のタスクの俯瞰ページとする
2. **ログのフォーマット** → 完全に自由テキスト。Claude Codeが状況に応じて自由に生成する。テンプレート制約なし
